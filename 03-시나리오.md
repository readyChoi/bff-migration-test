## H 시리즈: Hystrix 메모리 누적 관찰

Hystrix의 command metrics rolling window / RxJava 내부 버퍼로 인한 힙 메모리 증가 + Full GC 반복 패턴을 비교.

| Phase | 시간 | VU | 부하 패턴 | 관찰 포인트 |
|-------|------|-----|----------|------------|
| 1 | 0~30분 | 100 | 안정 부하 | baseline 힙 수준 측정 |
| 2 | 30~60분 | 300 | 고부하 (다양한 delay/size 혼합) | rolling window에 데이터 축적, 힙 증가 관찰 |
| 3 | 60~90분 | 5 | 거의 idle | 부하 제거 후 메모리가 반환되는지, GC가 힙을 줄이는지 |
| 4 | 90~120분 | 100 | 부하 재개 (Phase 1과 동일) | Phase 1 대비 힙이 높은지 (= 누적 있음) |

```bash
# k6와 힙 모니터링을 동시에 실행
./monitor-heap.sh 30 7200 &    # 30초 간격, 2시간
k6 run scenario-h-memory-leak.js
```

### 기대 결과

```
Java 8 + Hystrix:
  Phase 1: 힙 200MB
  Phase 2: 힙 350MB (증가)
  Phase 3: 힙 300MB (GC해도 완전히 안 떨어짐)  ← 누적
  Phase 4: 힙 320MB (Phase 1의 200MB보다 높음) ← 메모리 누수 증거

Java 25 + VT + R4j:
  Phase 1: 힙 150MB
  Phase 2: 힙 250MB (증가)
  Phase 3: 힙 160MB (거의 Phase 1 수준으로 복귀) ← 정상 반환
  Phase 4: 힙 150MB (Phase 1과 동일)             ← 누적 없음
```

> 수치는 예시. 실제 값은 heap size, 애플리케이션 크기에 따라 다름. 핵심은 **Phase 3에서 힙이 Phase 1 수준으로 돌아오는지**, **Phase 4에서 Phase 1보다 높은지**.

## H-2: Command Key 증가에 따른 메모리 누적

Hystrix는 command key별로 metrics를 **static map에 영구 보관**하므로, unique key가 늘수록 힙이 계속 증가한다. k6에서 path를 다양하게 만들어 key 수를 점진적으로 늘린다.

| Phase | 시간 | Unique Keys | 관찰 포인트 |
|-------|------|-------------|------------|
| 1 | 0~20분 | 100 | baseline 힙 |
| 2 | 20~40분 | 1,000 | 힙 증가 시작 |
| 3 | 40~60분 | 5,000 | 선형 증가 vs 폭증 |
| 4 | 60~80분 | 10,000 | 최대 key 수 |
| 5 | 80~100분 | 10,000 (새 key 없음) | 힙이 안정화되는지, GC로 줄어드는지 |

```bash
./monitor-heap.sh 15 6000 &
k6 run scenario-h2-key-growth.js
```

### 기대 결과

```
Java 8 + Hystrix:
  100 keys  → 힙 200MB
  1,000 keys → 힙 250MB
  5,000 keys → 힙 400MB
  10,000 keys → 힙 600MB+   ← key당 수 MB (thread pool + metrics)
  Phase 5    → 힙 600MB 유지 ← GC해도 안 줄어듦 (static map)

Java 25 + R4j:
  100 keys  → 힙 150MB
  1,000 keys → 힙 155MB
  5,000 keys → 힙 165MB
  10,000 keys → 힙 180MB    ← key당 수 KB (sliding window만)
  Phase 5    → 힙 180MB 유지 (크기 자체가 작아서 문제 안 됨)
```

> key당 메모리: Hystrix ~수 MB (thread pool + RxJava + metrics) vs R4j ~수 KB (sliding window only)

## 혼합 부하

k6 scenarios로 다른 특성의 요청을 동시에 실행.

```javascript
export const options = {
  scenarios: {
    fast_small: {
      executor: 'constant-vus', vus: 150, duration: '5m',
      exec: 'fastSmall',
    },
    slow_large: {
      executor: 'constant-vus', vus: 50, duration: '5m',
      exec: 'slowLarge',
    },
  },
};

export function fastSmall() {
  http.get(`${BFF}/mock/delay/100ms/size/1kb`);
}
export function slowLarge() {
  http.get(`${BFF}/mock/delay/2s/size/1mb`);
}
```

## 실행 우선순위

```
[필수]  A-1~A-4 — downstream latency 영향 (기본 검증)
[필수]  E-1, E-2 — 고동시성 확장성 (VT 증명)
[권장]  B-1~B-4 — response 크기 영향
[권장]  F-2, F-3 — CB 연계
[권장]  G-1 — 장시간 안정성
[선택]  C-1~C-4 — request 크기 영향
[선택]  E-3, E-4 — 극한 복합 조건
[권장]  H-1 — Hystrix 메모리 누적 관찰 (2시간, monitor-heap.sh 병행)
```

## 실행 방법

### 기본 스크립트 (환경변수로 시나리오 전환)

```javascript
// scenario.js
import http from 'k6/http';
import { check } from 'k6';

const BFF = __ENV.BFF_HOST || 'http://bff.test.svc';
const DELAY = __ENV.DELAY || '100ms';
const SIZE = __ENV.SIZE || '1kb';

export const options = {
  scenarios: {
    main: {
      executor: __ENV.EXECUTOR || 'constant-vus',
      vus: parseInt(__ENV.VUS || '200'),
      duration: __ENV.DURATION || '5m',
    },
  },
  thresholds: {
    http_req_duration: ['p(95)<3000'],
    http_req_failed: ['rate<0.05'],
  },
};

export default function () {
  const res = http.get(`${BFF}/mock/delay/${DELAY}/size/${SIZE}`);
  check(res, { 'status 200': (r) => r.status === 200 });
}
```

### 시나리오별 실행 예시

```bash
# A 시리즈 (delay 변화)
k6 run -e DELAY=10ms  scenario.js
k6 run -e DELAY=2s    scenario.js

# B 시리즈 (size 변화)
k6 run -e SIZE=1kb  scenario.js
k6 run -e SIZE=1mb  scenario.js

# E 시리즈 (고동시성)
k6 run -e VUS=500  scenario.js
k6 run -e VUS=1000 scenario.js

# 극한 부하 (ramping)
k6 run -e VUS=1000 -e EXECUTOR=ramping-vus scenario.js
```

### mock 응답 제어 3가지 방법

| 방법 | 사용 시점 | 예시 |
|------|----------|------|
| **path 파라미터** | 대부분 (사전 설정 불필요) | `/mock/delay/2s/size/1mb` |
| **PUT /config** | F-3 schedule처럼 동적 변경 필요 시 | `{"error_rate":100}` |
| **PUT /config/routes** | path prefix별 고정 config | `{"/slow":{"delay":"2s"}}` |

우선순위: path 파라미터 > routes > global config

### F-3 (CB lifecycle) 실행

```bash
# mock schedule 설정 후 k6 실행
curl -X PUT http://mock-downstream.test.svc/config/schedule \
  -d '{
    "phases": [
      {"at":"0s",   "config":{"delay":"100ms","error_rate":0,"payload_size":1024}},
      {"at":"120s", "config":{"delay":"100ms","error_rate":100,"error_code":500}},
      {"at":"240s", "config":{"delay":"100ms","error_rate":0,"payload_size":1024}}
    ]
  }'

# path 파라미터 없이 호출 → global config/schedule 적용
k6 run --vus 300 --duration 6m scenario-f3.js
```

### 시나리오 전환 절차

```bash
curl -X POST http://mock-downstream.test.svc/reset     # 초기화
curl http://mock-downstream.test.svc/metrics            # 이전 검증
k6 run --vus 10 --duration 2m scenario.js               # 워밍업
k6 run -e DELAY=500ms scenario.js                       # 본 테스트
kubectl cp bff-pod:/tmp/test.jfr ./results/scenario.jfr # JFR 수거
```
