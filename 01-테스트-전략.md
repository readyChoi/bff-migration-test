# 테스트 전략

## 목표

Java 8 + Hystrix vs Java 25 + Virtual Thread + Resilience4j — **동일 BFF 애플리케이션**의 성능/리소스 차이를 정량적으로 비교한다.

## A/B 비교 구조

```
[부하 생성기] ──→ [이미지 A: Java 8 + Hystrix]      ──→ [mock downstream]
[부하 생성기] ──→ [이미지 B: Java 25 + VT + R4j]    ──→ [mock downstream]
```

두 이미지를 **동일 인프라 스펙, 동일 mock downstream**에서 순차로 실행하여 비교한다.

## 테스트 방식

BFF 라우트 테이블에 `/mock` → `mock-downstream` 경로를 추가하고, k6가 `/mock/delay/{값}/size/{값}` 형태로 호출한다.

```
k6 ──→ BFF Pod ──(/mock/*)-→ mock-downstream Pod
```

- BFF 소스코드 변경 없음 (라우트 테이블만 추가)
- BFF의 요청 처리 파이프라인(인증 filter, thread pool 등)을 그대로 거침
- mock 응답 특성(delay, payload size, error rate)은 path 파라미터로 제어

## 비교 축

| 축 | 이미지 A (현재) | 이미지 B (목표) |
|----|----------------|----------------|
| Java | 8 | 25 |
| Thread 모델 | Platform Thread (thread pool) | Virtual Thread (per-request) |
| Circuit Breaker | Hystrix (Thread Pool 격리) | Resilience4j (Semaphore) |
| 동시 요청 처리 | pool size에 제한 | OS thread 수에 거의 무관 |
| 메모리 사용 | thread당 ~1MB 스택 + Hystrix pool | thread당 ~수 KB, 별도 pool 없음 |
| Context Switch | OS level + Hystrix pool 전환 | JVM level (cheap) |

> 개별 요인(JVM 업그레이드, VT, R4j) 분리는 하지 않는다. 실제 마이그레이션도 한번에 이루어지므로 **총합 효과**를 측정하는 것이 실용적이다.

## 테스트 단계

### Phase 1: Baseline 측정 (이미지 A)
- 현재 운영 환경과 동일한 설정으로 부하 테스트
- throughput, latency percentile, error rate, 리소스 사용량 기록

### Phase 2: 목표 구성 측정 (이미지 B)
- Java 25 + Virtual Thread + Resilience4j 구성으로 동일 시나리오 실행

### Phase 3: 비교 분석
- Phase 1 vs Phase 2 결과를 동일 메트릭 기준으로 비교
- 특히 **고동시성(high concurrency)** 구간에서의 차이에 집중

## 테스트 환경 조건

- BFF pod 스펙(CPU/Memory limit) 동일
- mock downstream의 응답은 path 파라미터로 고정
- JVM 옵션 명시적으로 기록 (GC, heap size 등)
- 워밍업 구간 포함 (최소 2~3분)
- 각 시나리오 최소 3회 반복, 중앙값 사용

## 테스트 범위

| 테스트 가능 (mock 방식) | 테스트 불가 (mock 방식) |
|------------------------|----------------------|
| downstream I/O blocking 시 thread 모델 차이 | BFF 내부 비즈니스 로직 (인증, 파일 처리 등) |
| Hystrix vs Resilience4j 오버헤드 | aggregation (여러 downstream 병렬 호출 + 조합) |
| 고동시성에서 thread pool 포화 vs VT 확장성 | |
