# 추후 고려 사항

현재 방식(`/mock` 라우트 + path 파라미터)으로 해결되지 않는 경우의 대안들.

---

## 1. BFF 내부 기능별 성능 테스트

### 현재 `/mock` 방식의 한계

`/mock` 경로는 BFF 입장에서 단순 프록시이므로, BFF 내부의 비즈니스 로직은 실행되지 않는다.

| 테스트 가능 (mock 방식) | 테스트 불가 (mock 방식) |
|------------------------|----------------------|
| downstream I/O blocking 시 thread 모델 차이 | 인증/토큰 검증 자체의 성능 |
| Hystrix vs Resilience4j 오버헤드 | 파일 업로드/다운로드 처리 |
| 고동시성에서 thread pool 포화 vs VT 확장성 | 메시지 발송 로직 |
| | aggregation (여러 downstream 병렬 호출 + 조합) |

### BFF 내부 기능까지 테스트하려면

BFF의 **실제 endpoint를 호출**해야 한다. 이때 downstream이 mock으로 가도록:

```yaml
# Istio VirtualService: BFF pod의 모든 outbound를 mock으로
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: catch-all-to-mock
  namespace: test
spec:
  hosts:
    - "*.prod.svc.cluster.local"
  http:
    - match:
        - sourceLabels:
            app: bff
      route:
        - destination:
            host: mock-downstream.test.svc.cluster.local
            port:
              number: 80
```

### 필요한 사전 조사

- BFF 실제 endpoint 목록 (actuator/mappings, Kiali, istio-proxy 로그)
- downstream 호출 시 사용하는 이름 형식 (short name vs FQDN)
- downstream 응답 JSON 구조 (mock의 `response_body` 설정에 필요)

## 2. Chaos Mesh 도입 (운영 안정성 테스트)

성능 비교가 끝난 후, **장애 대응 검증**을 별도로 할 때:

- BFF pod kill → k8s 재시작 동안 요청 유실량 측정
- Istio sidecar 장애 시 BFF 동작 확인
- BFF pod CPU throttling → Virtual Thread degradation 패턴 관찰
- 네트워크 완전 단절 → CB + timeout 조합 동작

상세는 [[99-QnA]] Q3 참조.
