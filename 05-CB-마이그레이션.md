# Circuit Breaker 마이그레이션: Hystrix → Resilience4j

## 배경

| 항목 | Hystrix | Resilience4j |
|------|---------|-------------|
| 상태 | 2018년 maintenance mode | 활발한 개발, VT 호환 |
| Thread 모델 | Thread Pool 격리 (별도 스레드) | Semaphore (호출 스레드에서 실행) |
| 설정 방식 | @HystrixCommand + YAML | @CircuitBreaker + YAML |
| 의존성 | hystrix-core + RxJava 1.x | resilience4j 단독 (경량) |
| 메트릭 | Hystrix Stream (자체 방식) | Micrometer 연동 (Prometheus 바로 연결) |

---

## Hystrix → Resilience4j 마이그레이션 장점

### 1. Thread Pool 제거 → 메모리/스레드 절감

```
[Hystrix]
downstream 10개 × coreSize 10 = 상시 100개 thread 유지
  + 요청 스레드 200개 (tomcat)
  = 총 300개 thread, 각 ~1MB 스택 = ~300MB

[Resilience4j + Virtual Thread]
Semaphore만 사용, 별도 thread pool 없음
  VT는 요청당 생성/소멸, 스택 ~수 KB
  = carrier thread 수십 개 + VT는 메모리 거의 안 씀
```

### 2. Hystrix의 메모리 누수 패턴 해소

Hystrix를 장시간 운행하면 메모리가 지속 증가하고 Full GC가 반복되는 현상이 있다. 원인:

| 원인 | 설명 |
|------|------|
| **HystrixCommandMetrics 누적** | 모든 command별로 rolling window 통계를 유지. command 종류가 많으면 메트릭 객체가 계속 쌓임 |
| **RxJava 1.x 내부 버퍼** | Hystrix는 RxJava 1.x 기반. Observable 체인에서 내부 버퍼가 GC되지 않는 경우 있음 |
| **HystrixRequestCache** | 요청 컨텍스트가 제대로 정리 안 되면 캐시 객체 누적 |
| **Thread Pool의 BlockingQueue** | thread pool의 대기 큐에 쌓인 작업이 제때 정리 안 됨 |
| **Hystrix Thread 자체** | coreSize만큼의 thread가 상시 존재. idle이어도 스택 메모리 점유 |

> Hystrix는 2018년 이후 maintenance mode라서 이런 문제가 수정되지 않음.

**Resilience4j는 이 문제들이 구조적으로 없다:**
- Thread Pool이 없음 → thread 메모리 누적 없음
- RxJava 의존성 없음 → 내부 버퍼 문제 없음
- 메트릭이 Micrometer 기반 → sliding window가 효율적으로 관리됨
- Semaphore는 상태를 거의 안 가짐 → GC 압력 최소

### 3. 의존성 경량화

```
[Hystrix]
hystrix-core → RxJava 1.x → archaius (Netflix config)
  + hystrix-javanica (어노테이션)
  + hystrix-metrics-event-stream (대시보드용)

[Resilience4j]
resilience4j-circuitbreaker (단독, 외부 의존성 없음)
  + resilience4j-spring-boot3 (Spring 연동)
  + resilience4j-micrometer (메트릭, optional)
```

Hystrix는 RxJava 1.x, archaius 등 Netflix OSS 의존성을 끌어오지만, Resilience4j는 Java 표준 라이브러리만 사용.

### 4. Micrometer/Prometheus 네이티브 연동

| Hystrix | Resilience4j |
|---------|-------------|
| Hystrix Stream → Turbine → Hystrix Dashboard (별도 인프라) | Micrometer → Prometheus → Grafana (이미 있는 인프라 활용) |
| 별도 `/hystrix.stream` endpoint 필요 | Spring Boot Actuator에 자동 등록 |

### 5. Virtual Thread 호환

| | Hystrix | Resilience4j |
|-|---------|-------------|
| Thread Pool 격리 | VT와 충돌 — Hystrix가 별도 platform thread로 실행하므로 VT 이점 무효화 | Semaphore — VT에서 그대로 실행, I/O시 자동 unmount |
| Semaphore 격리 | 지원하지만 thread pool이 기본 | 기본 방식 |

Hystrix의 thread pool 격리를 VT와 함께 쓰면, VT가 Hystrix pool의 platform thread 위에서 돌게 되어 **VT의 이점이 사라진다.**

---

### Thread 모델 차이 (성능 테스트의 핵심)

```
[Hystrix]
요청 스레드 → Hystrix Thread Pool(별도 스레드) → downstream 호출
  └─ 스레드 2개 소비, context switch 발생

[Resilience4j + Virtual Thread]
Virtual Thread → Resilience4j Semaphore(같은 스레드) → downstream 호출
  └─ 스레드 1개, context switch 없음, I/O시 자동 unmount
```

## Soak Test에서 확인할 포인트

G-1(30분 Soak Test)에서 아래를 비교한다:

| 메트릭 | Java 8 + Hystrix 예상 | Java 25 + VT + R4j 예상 |
|--------|----------------------|------------------------|
| 힙 사용량 추이 | 시간에 따라 증가 추세 (메트릭/RxJava 누적) | 안정적 유지 |
| Full GC 빈도 | 주기적 발생 (힙 압력) | 거의 없거나 매우 드묾 |
| GC pause 총합 | 높음 | 낮음 |
| 활성 thread 수 | 상시 높음 (Hystrix pool 유지) | 낮고 안정적 |

> 이 비교가 의미 있으려면 G-1의 duration을 30분 이상 (가능하면 1~2시간)으로 잡아야 한다. 메모리 누적은 시간이 지나야 드러남.

## 기능 검증 (마이그레이션 정합성)

| # | 테스트 케이스 | 기대 동작 |
|---|-------------|-----------|
| 1 | downstream 정상 응답 | 정상 통과 |
| 2 | downstream 타임아웃 | CB 타임아웃, fallback 반환 |
| 3 | downstream 연속 실패 | CB OPEN → fallback 즉시 반환 |
| 4 | CB OPEN → HALF_OPEN 전이 | 대기 시간 후 일부 요청 통과 |
| 5 | HALF_OPEN → CLOSED 복구 | 성공률 충족 시 정상 복귀 |
| 6 | Fallback 동작 | Hystrix와 동일한 응답 |
| 7 | Bulkhead (동시성 제한) | 초과 시 거절 |
| 8 | Retry 연동 | 재시도 후 성공 |

## 성능 시나리오

### CB-1: CB 오버헤드 비교

정상 상태에서 Hystrix vs R4j 순수 오버헤드. VU 200, 5분.

### CB-2: Failfast 성능

CB OPEN 상태에서 fallback 반환 속도. latency ≈ 0ms 되어야 함.

### CB-3: 고동시성 + CB

VU 100→500→1000 ramping. 이미지 A(Java 8+Hystrix) vs B(Java 25+VT+R4j) 비교.

### CB-4: Cascading Failure 방어

downstream 하나 장애 시 다른 downstream에 영향 주지 않는지 격리 검증.
- Hystrix: Thread Pool 격리 (서비스별 pool 분리)
- R4j: Bulkhead + Semaphore

## 설정 매핑

| Hystrix | Resilience4j |
|---------|-------------|
| `circuitBreaker.requestVolumeThreshold` | `minimumNumberOfCalls` |
| `circuitBreaker.errorThresholdPercentage` | `failureRateThreshold` |
| `circuitBreaker.sleepWindowInMilliseconds` | `waitDurationInOpenState` |
| `execution.isolation.thread.timeoutInMilliseconds` | `TimeLimiter.timeoutDuration` |
| `coreSize` (Thread Pool) | `Bulkhead.maxConcurrentCalls` |
| `@HystrixCommand(fallbackMethod=...)` | `@CircuitBreaker(fallbackMethod=...)` |
