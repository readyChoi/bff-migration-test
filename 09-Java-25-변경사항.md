# Java 8 → 25 주요 변경사항

Virtual Thread 외에 **성능 테스트 결과에 영향을 줄 수 있는** Java 8→25 변경사항 정리.

---

## G1 GC 개선

Java 8과 25 모두 G1을 사용하지만, G1 자체가 크게 개선되었다.

| 버전 | 개선 내용 |
|------|----------|
| Java 9 | G1이 기본 GC로 변경 (Java 8에서는 명시 필요) |
| Java 10 | **Full GC 병렬화** — Java 8 G1의 Full GC는 single-threaded, Java 10+는 parallel |
| Java 12 | **Abortable mixed collections** — mixed GC가 pause target 초과 시 중단 가능 |
| Java 12 | **Promptly return unused memory** — 유휴 시 OS에 메모리 반환 |
| Java 14 | **NUMA-aware allocation** — 멀티소켓 서버에서 메모리 할당 최적화 |
| Java 17 | Region 크기 자동 조정 개선 |
| Java 21 | Generational ZGC 도입 (G1은 아니지만 대안 선택지) |

### 테스트에서 확인할 포인트

- **같은 G1이라도 Java 25의 G1이 pause time이 짧을 수 있음** → GC pause 비교 시 이 차이 고려
- Full GC 발생 시 차이가 극적 (Java 8: single-thread → Java 25: parallel)
- 같은 heap size에서도 G1 동작이 다르므로, heap 설정을 동일하게 맞춰야 공정 비교

## JIT 컴파일러 개선

| 버전 | 개선 내용 |
|------|----------|
| Java 9 | Compact Strings (String 내부가 byte[] 기반으로 변경, 메모리 ~50% 절감) |
| Java 9 | Ahead-of-Time (AOT) 컴파일 실험적 도입 |
| Java 10 | Graal JIT 실험적 포함 |
| Java 11+ | C2 컴파일러 지속 최적화 (intrinsics, vectorization 등) |
| Java 16 | Vector API (incubator) — SIMD 연산 활용 가능 |
| Java 21+ | 다양한 intrinsic 최적화 누적 |

### 테스트에서 확인할 포인트

- Compact Strings: BFF가 JSON 파싱/직렬화를 많이 하면 **메모리 사용량 감소** 예상
- JIT 최적화 누적으로 **CPU bound 작업이 빨라질 수 있음** (인증 토큰 검증 등)
- 워밍업 패턴이 다를 수 있으므로 **워밍업 시간을 충분히** 확보

---

## 메모리 관련

| 버전 | 개선 내용 |
|------|----------|
| Java 9 | **Compact Strings** — Latin-1 문자열은 byte당 1바이트 (Java 8: 항상 2바이트) |
| Java 9 | String concatenation이 `invokedynamic` 기반으로 변경 → 더 효율적 |
| Java 11 | Epsilon GC (no-op GC, 벤치마크용) |
| Java 15 | Hidden classes (프레임워크의 동적 클래스 생성 효율화) |

### Compact Strings의 영향

BFF가 JSON을 많이 다루면 (ASCII 위주), Java 25에서 String 메모리가 약 절반으로 줄어든다.

```
Java 8:  "hello" = char[5] = 10 bytes
Java 25: "hello" = byte[5] = 5 bytes (Latin-1 인코딩)
```

→ 힙 사용량 비교 시 **Java 25가 낮게 나올 수 있는 이유 중 하나**

## 네트워크/HTTP

| 버전 | 개선 내용 |
|------|----------|
| Java 9 | HTTP/2 Client (incubator) |
| Java 11 | **java.net.http.HttpClient** 정식 포함 — HTTP/2, 비동기, WebSocket |
| Java 11+ | TLS 1.3 기본 지원 |

### 테스트에서 확인할 포인트

- BFF가 downstream 호출 시 사용하는 HTTP 클라이언트에 따라 영향 다름
  - Apache HttpClient, OkHttp 등 외부 라이브러리 사용 시: Java 버전과 무관
  - RestTemplate/WebClient: Spring 버전에 따라 내부 구현 다를 수 있음
- TLS 1.3: handshake가 빨라짐 (1-RTT → 0-RTT 가능). Istio mTLS 환경에서 효과 있을 수 있음

## 컨테이너 관련

| 버전 | 개선 내용 |
|------|----------|
| Java 8u191 | 컨테이너 인식 기본 활성화 (`-XX:+UseContainerSupport`) |
| Java 10 | 컨테이너 메모리/CPU 제한 인식 개선 |
| Java 15 | cgroup v2 지원 (현재 환경은 v1이므로 해당 없음) |

> 현재 cgroup v1 환경이므로, Java 8u191 이후와 Java 25의 컨테이너 인식 차이는 크지 않다. 단, heap 자동 계산(`-XX:MaxRAMPercentage`) 동작은 확인 필요.

## 기타 성능 관련

| 버전 | 개선 내용 |
|------|----------|
| Java 10 | **AppCDS** (Application Class Data Sharing) — 기동 시간 단축 |
| Java 12 | Switch expressions |
| Java 14 | Records |
| Java 15 | Text blocks |
| Java 16 | Pattern matching for instanceof |
| Java 17 | Sealed classes |
| Java 21 | Virtual Threads, Structured Concurrency (preview), Scoped Values (preview) |

> Records, sealed classes 등은 코드 품질 개선이지 성능과는 무관. AppCDS는 기동 시간 단축에 효과적이나 런타임 성능에는 영향 없음.

---

## 요약: 성능 테스트 결과에 영향을 줄 수 있는 변경점

| 영역 | 예상 영향 | 비교 시 주의 |
|------|----------|-------------|
| G1 GC 개선 (Full GC 병렬화 등) | GC pause 감소 | 같은 G1이라도 Java 25가 유리 |
| Compact Strings | 힙 메모리 감소 | Java 25에서 메모리가 낮게 나오는 이유 중 하나 |
| JIT 최적화 누적 | CPU bound 작업 소폭 개선 | 워밍업 시간 충분히 확보 |
| TLS 1.3 | handshake 시간 감소 | Istio mTLS 환경에서 효과 가능 |
| Virtual Thread | I/O blocking 시 thread 모델 차이 | **핵심 측정 대상** |

> 테스트 결과에서 성능 개선이 나왔을 때, 그게 **전부 VT 덕분인지 아니면 G1/JIT/Compact Strings 개선도 기여한 건지**는 분리할 수 없다. 이건 처음부터 "총합 효과"를 측정하기로 했으므로 문제없다.
